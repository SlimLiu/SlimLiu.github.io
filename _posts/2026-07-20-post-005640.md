# 走向自进化 Agent：真正关键的不是更新，而是表示与评价

我们常常用“能不能完成任务”来评价一个 Agent：能不能写完代码、打赢一局牌、调用正确的工具，或者给出一个看起来合理的答案。

一个更重要的问题是：**任务结束以后，Agent 留下了什么？**

如果它只是完成了一次任务，却没有沉淀经验、修正策略，也无法判断自己的改变究竟是进步还是退步，那么无论单次表现多么惊艳，它都还不能被称为一个会学习的系统。

## 从“越打越强”到“如何学习”

讨论最初来自一个很具体的问题：如何让德州扑克 Agent 越打越强？

顺着这个问题往下走，很快就会遇到一组更底层的问题：

- Agent 能否像人一样从经验中学习？
- Agent 的哪些部分可以持续演化？
- 没有可靠的评价机制，是否根本谈不上自进化？
- Skill 和 Memory 应该采用什么表示，才便于系统自动读取、修改和验证？

这时，研究对象变成了一个更一般的问题：**如何构造一个能够持续积累经验、修改自身，并验证修改效果的 Agent？**

## 自进化是一个闭环

一个初步的自进化闭环可以写成：

```text

Environment

    ↓

Experience

    ↓

Memory

    ↓

Reflection

    ↓

Skill Update

    ↓

Evaluation

    ↓

Selection

    ↓

下一轮 Experience

```

这个闭环中的每一步都有明确作用：

- **Environment** 提供任务、反馈和约束；
- **Experience** 记录一次真实交互中发生了什么；
- **Memory** 保存值得长期复用的信息；
- **Reflection** 从结果中提炼规律、错误与改进方向；
- **Skill Update** 把反思转化为可执行的策略变化；
- **Evaluation** 判断变化是否真的有效；
- **Selection** 决定保留、回滚还是继续修改。

这意味着，自进化不能只是“让模型改写自己的 Prompt”或“把上一轮内容存进记忆”。如果没有后续评价和选择，更新只是在制造变化，而变化本身并不等于进步。

## Evaluation 比 Update 更重要

这是这次讨论中最重要的判断之一。

我们很容易把注意力放在 Update 上：如何更新 Prompt，如何重写 Skill，如何压缩 Memory，如何微调参数。但真正困难的问题其实是：**系统凭什么知道这次更新更好了？**

如果奖励信号不可靠，Agent 就可能：

- 把一次偶然成功误认为稳定能力；
- 为了短期指标牺牲长期表现；
- 在反复更新中遗忘原本有效的策略；
- 用更复杂的规则换来不可验证的“看起来更聪明”。

因此，自进化系统首先需要的不是更激进的更新算法，而是一套可信的评价机制。它至少要回答四个问题：

1. **评价什么？** 是任务成功率、长期收益、鲁棒性，还是成本与速度？
2. **和谁比较？** 是和上一版本、固定基线，还是同一 Base Model 下的对照 Agent 比较？
3. **在什么数据上比较？** 如何避免 Agent 只记住已经见过的任务？
4. **多大差异才算进步？** 如何区分真实提升与随机波动？

只有当 Evaluation 足够稳定，Selection 才有意义；只有 Selection 有意义，Update 才能进入一个正向循环。

## Representation 可能比更新算法更重要

第二个关键判断是：**Agent 如何表示自己的知识，可能比它如何更新知识更重要。**

一种表示是否适合自进化，至少取决于它是否容易被读取、定位、修改、组合、回滚和评价。

不同类型的知识，可能天然适合不同表示：

- **Markdown / 自然语言**：适合原则、解释、经验总结和人机共同维护；
- **YAML / 结构化记录**：适合带字段的 Memory、规则配置和可追踪元数据；
- **程序代码**：适合确定性流程、工具封装和可测试的执行策略；
- **图结构**：适合表达概念之间的关系、依赖与路径；
- **参数更新（如 LoRA）**：适合难以显式写成规则、却能从数据中吸收的模式。

这也说明，我们不一定应该追求一种统一表示。Memory、Skill、Planner 和 Evaluator 承载的是不同性质的知识，它们完全可能需要不同的表示方式和更新方法。

更值得追问的不是“哪种表示最强”，而是：**哪种表示最适合某一模块持续演化，并且能被可靠评价？**

## 德州扑克：理想实验场，还是评价陷阱？

德州扑克很适合研究 Agent，因为它同时具有三个重要特征：

- 长期决策（Long-horizon）；
- 不完全信息（Imperfect Information）；
- 收益可以量化。

但它也暴露了 Evaluation 的核心难题：运气会对短期结果产生巨大影响。

如果只看一局输赢，Agent 很容易从噪声中学到错误规律。因此，扑克 Agent 的评价必须建立在大量对局之上，并结合 Elo、平均收益和置信区间等统计指标。研究重点也不只是“是否赢了”，而是策略在不同对手、不同局面和足够长时间范围内是否稳定提升。

换句话说，德州扑克既是研究长期学习的好环境，也是一个提醒：**奖励可量化，不代表评价就天然可靠。**

## 为什么 Coding Agent 更容易研究自进化

与扑克相比，Coding Agent 拥有更成熟的评价基础设施：

- SWE-bench 等 Benchmark 提供标准化任务；
- 单元测试能够给出相对明确的 Pass / Fail Reward；
- GitHub Issue 与 Pull Request 提供大量真实工作流；
- 代码修改天然适合版本比较、回滚和消融实验。

这让 Coding Agent 更容易形成完整的“修改—测试—比较—选择”闭环。一个 Skill 是否有效，可以放到一组任务上复测；一次规划策略更新是否带来提升，也可以和旧版本进行可重复的对照。

因此，如果目标是先验证 Self-Evolving Agent Framework，Coding Agent 可能是比德州扑克更容易起步的研究对象。扑克环境则更适合进一步检验系统在长期决策、不完全信息和高噪声奖励下是否仍然有效。

## 下一步真正值得研究的问题

围绕这个框架，还有一组尚未解决的问题：

1. Skill 能否完全自动生成、修改和淘汰？
2. Memory 与 Skill 能否采用相似的双循环演化框架？
3. 不同模块是否应该使用不同的知识表示？
4. 什么样的表示最适合持续修改、组合与回滚？
5. 如何设计稳定、可泛化且不易被投机的 Evaluation？
6. 如何证明性能提升来自 Agent 的演化，而不是更强的 Base Model？
7. 能否构造一个适用于多任务的通用 Self-Evolving Agent Framework？

这些问题最终都指向同一个研究目标：把 Agent 从一次性执行器，变成一个能够积累经验、修正自身，并对进步负责的学习系统。

## 结语

过去我们问的是：怎样让 Agent 把任务做对？

接下来更值得问的是：怎样让 Agent 在做完任务以后，知道自己学到了什么；怎样把这种学习写回自身；又怎样证明下一次真的做得更好？

从这个角度看，自进化的关键也许并不是 Update 本身，而是它两侧的两个基础设施：

> **Representation 决定 Agent 能改变什么，Evaluation 决定这种改变是否值得保留。**

只有当二者都成立，Agent 的每一次行动才不只是一次任务执行，也可能成为下一次进步的起点。
